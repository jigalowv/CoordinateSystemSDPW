## Архітектура рішення

```
┌──────────────────────────────────────────────────────┐
│                                                      │
│   GPS Емулятор (Docker, :4001)                       │
│   WebSocket + REST /config                           │
│                                                      │
└────────────┬─────────────────────────────────────────┘
             │ WebSocket (дані супутників)
             ▼
┌──────────────────────────────────────────────────────┐
│   Фронтенд (index.html)                              │
│                                                      │
│  • Підключається до WS емулятора                     │
│  • Буферизує дані (300 мс вікно)                     │
│  • Розраховує відстані через затримку сигналу        │
│  • POST /calculate → FastAPI бекенд                  │
│  • Відображає результати на графіку (Chart.js)       │
│                                                      │
└────────────┬─────────────────────────────────────────┘
             │ HTTP POST /calculate
             ▼
┌──────────────────────────────────────────────────────┐
│   Python FastAPI бекенд (:8000)                      │
│                                                      │
│  • Аналітичний метод (система лінійних рівнянь)      │
│  • Чисельний метод (scipy.optimize.minimize)         │
│  • Повертає обидва результати у JSON                 │
│                                                      │
└──────────────────────────────────────────────────────┘
```

**Перевага варіанту 4:** Фронтенд отримує "сирі" дані через WebSocket напряму, а Python бекенд виступає stateless "калькулятором" — приймає координати супутників і відстані, повертає розраховані координати об'єкта. Не потрібно керувати постійними WebSocket-з'єднаннями на сервері.

## Запуск — крок за кроком

### Крок 1 — Запуск GPS емулятора (Docker)

```bash
# Завантаження образу
docker pull iperekrestov/university:gps-emulation-service

# Запуск контейнера
docker run --name gps-emulator -p 4001:4000 iperekrestov/university:gps-emulation-service
```

Перевірте, що емулятор працює:
```bash
# Має відповісти 200 OK або JSON
curl http://localhost:4001
```

### Крок 2 — Запуск Python бекенду

```bash
# Перейти до папки бекенду
cd gps_lab/backend

# Встановити залежності
pip install -r requirements.txt

# Запустити сервер
uvicorn main:app --host 0.0.0.0 --port 8000 --reload
```

Перевірте, що бекенд працює:
```bash
curl http://localhost:8000/health
# {"status":"healthy"}
```

Документація API доступна за адресою: http://localhost:8000/docs

### Крок 3 — Відкрити фронтенд

Просто відкрийте файл у браузері:
```bash
# macOS
open gps_lab/frontend/index.html

# Linux
xdg-open gps_lab/frontend/index.html

# Windows
start gps_lab/frontend/index.html
```

> **Примітка:** Якщо браузер блокує fetch-запити через CORS (файл відкрито як `file://`), запустіть локальний HTTP-сервер:
> ```bash
> cd gps_lab/frontend
> python -m http.server 3000
> # Відкрийте http://localhost:3000
> ```

## Як користуватись додатком

### Підключення

1. Переконайтесь, що поле **WebSocket емулятора** містить: `ws://localhost:4001`
2. Переконайтесь, що поле **API бекенду** містить: `http://localhost:8000`
3. Натисніть кнопку **«Підключитись»**
4. Індикатор у правому верхньому куті стане зеленим ✅

### Спостереження за даними

- **Графік** оновлюється в реальному часі
- **Сині квадрати** — поточне положення супутників
- **Зелені кола** — положення об'єкта (аналітичний метод)
- **Помаранчеві ромби** — положення об'єкта (чисельний метод)
- **Пунктирні лінії** — траєкторії (останні 50 точок)

### Зміна параметрів емулятора

1. Використовуйте **повзунки** у бічній панелі для налаштування:
   - Розмір зони (50–500 км)
   - Частота повідомлень (1–10 msg/s)
   - Швидкість супутників (10–500 км/г)
   - Швидкість об'єкта (1–200 км/г)
2. Натисніть **«Застосувати параметри»**

Або через `curl`:
```bash
curl -X POST http://localhost:4001/config \
  -H "Content-Type: application/json" \
  -d '{"satelliteSpeed": 150, "objectSpeed": 30}'
```

### Панель статистики

| Показник | Значення |
|----------|----------|
| Пакетів | Загальна кількість отриманих повідомлень |
| Супутників | Кількість активних супутників у буфері |
| Похибка (м) | Відстань між аналітичним та чисельним результатами |
| Затримка (мс) | Час обробки одного кадру бекендом |

## Опис алгоритмів

### Розрахунок відстані

Відстань від об'єкта до супутника розраховується через затримку сигналу:

```
відстань = (receivedAt − sentAt) × швидкість_світла
```

де `швидкість_світла ≈ 299 792.458 км/мс`

### Аналітичний метод (трилатерація)

Рівняння кола для кожного супутника:
```
(x − xᵢ)² + (y − yᵢ)² = rᵢ²
```

Віднімаючи рівняння першого супутника від другого та третього, отримуємо **лінійну систему**:

```
2(x₂−x₁)·x + 2(y₂−y₁)·y = r₁²−r₂² + x₂²−x₁² + y₂²−y₁²
2(x₃−x₁)·x + 2(y₃−y₁)·y = r₁²−r₃² + x₃²−x₁² + y₃²−y₁²
```

Розв'язуємо методом `numpy.linalg.solve` (метод Гаусса). Результат — точні координати для ідеальних даних.

### Чисельний метод (scipy.optimize.minimize)

Визначаємо **функцію втрат** — суму квадратів різниць між виміряними та теоретичними відстанями:

```
L(x, y) = Σᵢ ( √((x−xᵢ)² + (y−yᵢ)²) − rᵢ )²
```

Мінімізуємо `L(x, y)` методом **L-BFGS-B** (квазі-ньютонівський метод обмеженої пам'яті):
- Початкове наближення: центроїд позицій супутників
- Метод стійкий до шуму у вхідних даних
- Параметри: `ftol=1e-12`, `gtol=1e-10`, `maxiter=1000`

## API бекенду

### `POST /calculate`

**Тіло запиту:**
```json
{
  "satellites": [
    {
      "id": "uuid",
      "x": 50.0,
      "y": 120.3,
      "distance": 45.7
    }
  ]
}
```

**Відповідь:**
```json
{
  "analytical": { "x": 98.5, "y": 147.2 },
  "numerical":  { "x": 98.6, "y": 147.1 },
  "error": null
}
```

> Якщо `analytical = null` — система рівнянь вироджена (супутники на одній прямій).

### `GET /health`

```json
{ "status": "healthy" }
```

## Технічні деталі

### Стек технологій

| Компонент | Технологія |
|-----------|-----------|
| Бекенд | Python 3.10+, FastAPI, scipy, numpy |
| Фронтенд | Vanilla HTML/CSS/JS (без збірки) |
| Графік | Chart.js 4.4 (CDN) |
| Емулятор | Docker (готовий образ) |

### Буферизація даних на фронтенді

Фронтенд збирає повідомлення від супутників у **вікні 300 мс**, потім формує один "кадр" і відправляє на бекенд. Це дозволяє зібрати дані від усіх супутників перед розрахунком, навіть якщо повідомлення надходять не одночасно.

### CORS

Бекенд дозволяє запити з будь-якого джерела (`allow_origins=["*"]`), що спрощує розробку. У продакшені слід обмежити до конкретного домену.

## Можливі помилки

| Помилка | Причина | Рішення |
|---------|---------|---------|
| "Помилка WebSocket з'єднання" | Емулятор не запущений | Запустіть Docker контейнер |
| "Помилка запиту до бекенду" | FastAPI не запущений | Запустіть `uvicorn main:app` |
| "Потрібно щонайменше 3 супутники" | Мало даних | Зачекайте кілька секунд |
| analytical = null | Вироджена система | Нормально, використовується чисельний результат |
| CORS помилка в браузері | Файл відкрито через file:// | Використайте `python -m http.server` |
